## P0 修复清单

### P0-0：禁止客户端覆盖保留 metadata 字段（order_no/user_id）

**位置**: `src/app/api/payment/checkout/route.ts:216-221`

**问题**: checkout 接口将客户端 `metadata` 通过 `...(metadata || {})` 合并在末尾，可能覆盖服务端写入的 `order_no` / `user_id`。由于 webhook 侧依赖 `metadata.order_no` 定位订单，这会导致“支付结果绑定到错误订单”的风险（错发权益/错发 credits）。

**修复方案**:
```typescript
// 让服务端保留字段后置写入，避免被客户端覆盖
metadata: {
  ...(metadata || {}),
  app_name: configs.app_name,
  order_no: orderNo,
  user_id: user.id,
}
```

**执行步骤**:
1. 调整 `checkout/route.ts` 中 `checkoutOrder.metadata` 的合并顺序（服务端字段后置）
2. （可选）在 `PaymentCheckoutBodySchema` 层拒绝 `metadata` 中包含 `order_no` / `user_id`

### P0-1：订阅续费幂等缺失

**位置**: `src/app/api/payment/notify/[provider]/route.ts:100-131`

**问题**: renewal 入口缺少 transactionId 缺失保护与幂等短路；虽 `updateSubscriptionInTransaction` 内部会按 `(transactionId, paymentProvider)` 查重，但当 `transactionId` 缺失或并发竞态时仍可能重复入库/重复发放 credits。

**Context7 最佳实践**:
> "Handle duplicate events: You can guard against duplicated event receipts by logging the event IDs you've processed, and then not processing already-logged events."
> 
> "To identify these duplicates, use the ID of the object in `data.object` along with the `event.type`."

**修复方案**:
```typescript
// 方案 A（推荐）：使用 transactionId（Stripe: invoice.id，即 data.object.id）作为幂等键
// 关键前提：renewal 必须有稳定的 transactionId，否则拒绝处理（避免重复发放）
const transactionId = session.paymentInfo?.transactionId;
if (!transactionId) {
  throw new BadRequestError('transaction id not found');
}

const existingOrder = await findOrderByTransactionId({ provider, transactionId });
if (existingOrder) return jsonOk({ message: 'already processed' });
```

**执行步骤**:
1. 在 `src/shared/models/order.ts` 添加 `findOrderByTransactionId` 函数
2. 在 `notify/route.ts` 的 `PAYMENT_SUCCESS` 分支添加幂等检查
3. 添加调试日志记录重复事件
4. （强烈建议）补充 DB 级唯一约束：`order(transaction_id, payment_provider)`，消除并发竞态导致的重复入库/重复发放

### P0-3：金额防篡改验证

**位置**: `src/shared/services/payment/flows.ts:44-143`

**问题**: `handleCheckoutSuccess` 信任 session 中的金额，未与 order 原始金额交叉校验。

**Context7 最佳实践**:
> "Calculate the order total on the server to prevent people from directly manipulating the amount on the client"
> 
> "Recalculate amounts on your server. Don't trust client-provided prices or totals."
> 
> "Security best practices: Recalculate amounts on your server. Don't trust client-provided prices or totals."

**修复方案**:
```typescript
// 校验支付金额与订单金额（注意：Stripe 使用最小货币单位，例如 USD 分）
// 策略：禁止“少付”（防错单/防篡改）；允许“多付”（税费/尾差）
const paidAmount = session.paymentInfo?.paymentAmount;
const paidCurrency = session.paymentInfo?.paymentCurrency;
if (paidAmount == null || !paidCurrency) throw new Error('invalid payment info');

if (paidCurrency.toLowerCase() !== order.currency.toLowerCase()) {
  throw new Error('payment currency mismatch');
}

const discount = session.paymentInfo?.discountAmount ?? 0;
const expectedMinAmount = Math.max(0, order.amount - discount);
if (paidAmount < expectedMinAmount) {
  throw new Error('payment amount mismatch');
}
```

**执行步骤**:
1. 在 `handleCheckoutSuccess` 中添加金额校验逻辑
2. 统一使用“最小货币单位”校验，并支持 `discountAmount`（禁止少付，允许多付）
3. 记录异常日志用于审计
