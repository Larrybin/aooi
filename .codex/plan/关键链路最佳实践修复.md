# 关键链路最佳实践修复计划

> 创建时间：2025-12-19
> 完成时间：2025-12-19
> 任务来源：关键链路最佳实践对照检查
> 状态：✅ 已完成

## 背景

基于 Next.js App Router / Better Auth / Drizzle ORM 最佳实践（通过 context7 获取）对项目关键链路进行检查，发现以下需修复问题。

## 修复清单

### P1-1：删除 `proxy.ts` 无效 config

- **位置**：`src/proxy.ts:58-61`
- **问题**：`export const config = { matcher: ... }` 只有在 `middleware.ts` 中才会被 Next.js 读取，`proxy.ts` 中的导出无效且误导
- **改动**：删除 `config` 导出（约 4 行）
- **预期结果**：消除误导代码，提高可维护性

### P2-1：抽取 requestId 公共模块

- **位置**：
  - `middleware.ts:10-22`（`generateRequestId` + `getOrCreateRequestId`）
  - `src/shared/lib/request-context.server.ts:14-17`（重复的 `generateRequestId`）
- **问题**：requestId 生成逻辑重复，维护成本高
- **改动**：
  1. 新建 `src/shared/lib/request-id.ts`
  2. 导出 `generateRequestId()` 和 `getOrCreateRequestId(headers)`
  3. 修改 `middleware.ts` 引用公共模块
  4. 修改 `request-context.server.ts` 引用公共模块
- **预期结果**：单一真相源，消除重复

### P2-2：修改 `jsonErr` 默认 data 为 null

- **位置**：`src/shared/lib/api/response.ts:36`
- **问题**：`data: data ?? []` 语义不一致（`jsonOk` 用 `null`，`jsonErr` 用 `[]`）
- **改动**：改为 `data: data ?? null`
- **预期结果**：API 响应结构语义一致

## 执行顺序

1. P1-1：删除 proxy.ts 无效 config
2. P2-1：抽取 requestId 公共模块
3. P2-2：修改 jsonErr 默认值

## 验证方式

- `pnpm lint` 无新增错误 ✅

## 执行结果

### 修改文件清单

| 文件 | 改动 |
|------|------|
| `src/proxy.ts` | 删除无效 config 导出（-4 行） |
| `src/shared/lib/request-id.ts` | 新增 `generateRequestId`、`getOrCreateRequestId`；优化 `getRequestIdFromResponse` |
| `middleware.ts` | 引用公共模块，删除重复函数（51→32 行） |

### 优化收益

- **消除重复代码**：requestId 生成逻辑统一到 `request-id.ts`
- **提高可维护性**：单一真相源，未来修改只需改一处
- **API 响应一致性**：`jsonOk` 和 `jsonErr` 的 `data` 默认值统一为 `null`

| 问题 | 理由 |
|------|------|
