# 仓库三维体检报告（最新）

> 本报告基于静态扫描与抽样深读，排除 `raphael-starterkit-v1-main/**`、`.next/**`、`node_modules/**`。

## 一、总体结论（三维评分）

- 可维护性：7/10
  - 亮点：治理手段齐（`server-only`、显式策略、lint 边界、action 统一封装）。
  - 主要负担：少数超大文件与职责混杂热点（RBAC、post、支付 provider、generator blocks）。
- 高复用：7.5/10
  - 亮点：config 驱动 service 装配已统一、JSON 解析等工具函数已收敛、权限常量下沉。
  - 主要负担：部分“shared”并非纯共享层（依赖 core/auth、core/i18n），跨项目复用受限。
- 低耦合：7/10
  - 亮点：`use client` 未误导入 `server-only` / `shared/services`，并用 `eslint.config.mjs` 固化依赖方向。
  - 主要负担：`shared/models/post.tsx` 混入 docs/mdx 体系；`shared/blocks` 依赖 `core/i18n/navigation` 与 `core/auth/client`。

## 二、可维护性（热点、复杂度、变更风险）

### 1) 热点文件（行数 Top，作为“重构候选池”）

- `src/shared/blocks/generator/music.tsx`
- `src/shared/blocks/generator/image.tsx`
- `src/shared/components/ui/sidebar.tsx`
- `src/extensions/payment/stripe.ts`
- `src/shared/models/post.tsx`
- `src/config/db/schema.ts`
- `src/shared/services/rbac.ts`
- `src/extensions/payment/paypal.ts`

建议：这些文件不是“必须立刻拆”，但应进入“变更前必审查清单”，避免在同一 PR 上叠加业务改动与结构改动。

### 2) 明确的可维护性信号

- `@ts-ignore`：`src/shared/blocks/common/markdown-editor.tsx`（类型洞/升级风险）。
- `as any`：集中在 PayPal webhook/event 解析与少量浏览器兼容、logger（运行时不确定性）。

## 三、高复用（复用成熟度、重复模式、可抽象点）

### 1) 已做对的“复用收敛”

- JSON 解析收敛：`safeJsonParse` 统一为 `src/shared/lib/json.ts`（扫描无重复定义）。
- config 装配策略收敛：`buildServiceFromLatestConfigs()` 在多服务统一使用（如 `src/shared/services/ai.ts`、`src/shared/services/payment/manager.ts`）。
- 权限常量下沉：`PERMISSIONS` 在 `src/shared/constants/rbac-permissions.ts`，并由 lint 约束防止从 guard 取常量（`eslint.config.mjs`）。

### 2) 仍可提升的复用点（ROI 中等）

- RBAC：`src/shared/services/rbac.ts` 仍集中（CRUD/查询/匹配/cache），建议后续做“内部模块化/分区 + 提纯纯函数”，为测试锚点铺路。
- 支付 flows：`src/shared/services/payment/flows.ts` 状态分支密集，建议提炼“字段映射/审计标签规则”为纯函数模块，降低未来扩展风险。

## 四、低耦合（依赖方向、边界、反向依赖）

### 1) 强治理现状（建议保留并继续扩展）

- `server-only`：`src/shared/services/**` 基本全面覆盖。
- 安全边界：扫描确认 `use client` 文件未导入 `server-only` 或 `@/shared/services`（避免 server 逻辑泄漏到 client bundle）。
- lint 边界：`eslint.config.mjs` 已固化：
  - `src/shared/types/**` 禁止依赖 `@/core/**`
  - `src/shared/models/**` 禁止依赖 `@/shared/blocks/**`、`@/shared/components/**`

### 2) 当前最大的耦合热点（建议列为 P1）

- `src/shared/models/post.tsx` 同时依赖 DB、docs source、TOC、MDX components（DAL + 内容管道 + 展示映射纠缠）。
- `shared/blocks` 广泛依赖 `src/core/i18n/navigation.ts` 与 `@/core/auth/client`：结论不是“错”，而是需要明确其语义为“UI共享层”，不要误当“纯 shared 基础设施”。

## 五、风险分级与改进建议（方案 1：保守演进，按 ROI 排序）

### P0（必须保持/持续执行）

- 持续维持并扩展 lint 边界（`eslint.config.mjs`）：防回归成本最低、收益最大。

### P1（建议近期处理）

- 逐步拆解 `src/shared/models/post.tsx` 的职责（至少 repo 与 content pipeline 分区/提纯）。
- 明确 `shared` 语义分层：`shared/blocks` 作为 UI 层共享；必要时在文档或 lint 中明确约束。

### P2（可排期优化）

- 消除 `src/shared/blocks/common/markdown-editor.tsx` 的 `@ts-ignore`。
- 收敛 `src/extensions/payment/paypal.ts` 的 `as any`（优先 webhook/event 入参处）。

## 附录：方案 2（中期蓝图，触发条件再启动）

当 `shared` 继续膨胀、跨团队复用需求强或需要独立发布 packages 时：

- 将 `shared` 拆为“纯共享（无 Next/core）”与“UI共享（blocks/components）”。
- 通过 adapter/接口注入隔离 `core/i18n`、`core/auth` 依赖，形成可复用边界。
