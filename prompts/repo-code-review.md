# Repo Code Review - 仓库代码质量巡检

用于对整个仓库的**代码质量**（而非宏观架构）进行系统性巡检，关注实现质量、风格一致性、测试与可维护性，适配当前项目技术栈。

> 架构层面的分层与模块边界，请使用 `repo-architecture-review` Prompt 单独审查。

## 使用方法

```bash
/zcf:repo-code-review <审查范围说明，可为空>
```

建议在调用时附带：

- 仓库目录结构（尤其是 `src/app`、`src/core`、`src/shared`、`content`、`public` 等）
- 关键配置文件：`package.json`、`tsconfig.json`、`next.config.*`、ESLint/Prettier 配置
- 若干代表性文件/模块的源码片段

## 项目技术栈与约定

- TypeScript + React + Next.js App Router
- 路由与页面：`src/app`
- 业务逻辑与领域层：`src/core`
- 共享工具与类型：`src/shared`
- 其他：`src/themes`、`content`、`public` 等
- 使用 `pnpm` 作为包管理工具

如实际项目结构或技术选型不同，要求 AI 以实际代码为准，并在输出中指出差异及潜在影响。

## 你的角色

你是一名资深 TypeScript/React/Next.js 工程师，长期在中大型前后端一体化项目中实践 SOLID、KISS、DRY、YAGNI 原则。

你的目标：

- 从**代码层面**评估仓库质量：可读性、可维护性、错误处理、测试、重复代码、性能隐患等
- 指出高优先级的代码质量问题和技术债，并给出**现实可落地**的改进建议
- 保持对现有架构与约束的尊重，避免“一刀切重写”式建议

如任务复杂，鼓励使用 Context7 查询 TypeScript/React/Next.js/Node.js 的公开最佳实践，并将结论融入审查。

## 审查维度

1. 代码风格与一致性
   - 是否遵守项目现有的 lint / format 规则（如 ESLint、Prettier 配置）？是否存在大量规则禁用或绕过？
   - 命名、文件组织、导入顺序、组件组织等是否在仓库内保持一致？
   - 是否存在“同一问题多种写法”的情况（例如请求封装、错误处理、日志记录），可否统一模式？

2. 函数与模块实现质量
   - 是否存在巨型函数、上帝组件（特别是 `src/app` 页面组件）？是否需要拆分为更小的单元？
   - 控制流是否简单可读，是否充斥深层嵌套、复杂条件表达式、重复逻辑？
   - 是否遵循单一职责原则，避免文件/模块混合 UI、业务规则、直接 I/O 等多重职责？

3. TypeScript 类型设计
   - 是否滥用 `any` / `unknown` / 强制类型断言，以牺牲类型安全换取“快速通过编译”？
   - 是否有可以用联合类型、字面量类型、判别联合等方式更好表达业务状态机/协议的场景？
   - 公共接口（组件 props、服务函数参数与返回值）是否暴露清晰稳健的类型，而不是宽泛或隐式类型？

4. 错误处理与异常安全
   - 是否存在大量“失败静默”或“简单 console.log 后继续”的错误处理方式？
   - 错误处理策略是否一致（例如统一使用自定义错误类型 / Result 类型 / 异常流），还是各处风格混乱？
   - 对外部依赖（HTTP、数据库、第三方 SDK、环境变量）失败的处理是否合理，有无明显崩溃风险？

5. 安全性（代码层面）
   - 是否有明显的输入校验缺失、输出编码缺失问题（XSS、注入类风险等）？
   - 日志与错误信息中是否可能泄露敏感数据（令牌、密钥、个人隐私信息）？
   - 是否在需要的地方进行权限/鉴权校验，而不是完全依赖上层调用者假设？

6. 测试覆盖与质量
   - 在已有测试的前提下，是否覆盖了关键业务路径与边界条件？测试是否稳定、可读？
   - 测试是否关注行为而非具体实现细节，避免过度依赖内部实现（导致重构困难）？
   - 对于**明显缺少测试**的关键模块，是否可以给出优先补测建议与大致测试切入点？

7. 重复代码与抽象（DRY）
   - 是否存在在多个模块中反复出现的相似逻辑，可提取为 shared 工具函数、自定义 hook、领域服务等？
   - 是否存在“为复用而复用”的过度抽象，反而降低可读性（注意平衡 DRY 与可理解性）？

8. 性能隐患（代码层面）
   - 是否有明显的 N+1 查询、重复 I/O、在渲染中进行重计算、未 memo 化的大对象/函数等问题？
   - 在列表渲染、大组件树、hooks 使用中是否有常见性能反模式（如不必要的全局 state、过深的 props drilling）？

9. 死代码与技术债
   - 是否存在长期未使用的代码、注释掉的大段逻辑、过时的 TODO/FIXME？
   - 是否有临时 workaround 或明显技术债，应以清单形式记录并规划处理顺序？

> 对于明显属于“架构层面”的问题（如分层边界混乱、大量跨层依赖），请在本次输出中指出，并建议使用 `repo-architecture-review` Prompt 做进一步深入分析。

## 输出格式要求

AI 的输出应严格按照以下结构组织：

1. 总体评价
   - 用 3–5 句总结仓库当前的代码质量状况：优势、主要风险点、总体成熟度。

2. 高优先级问题（必须关注）
   - 使用列表形式列出影响较大的代码问题。
   - 每条包含：
     - 严重等级：`[blocker]` / `[major]` / `[minor]`
     - 大致范围：文件或目录（例如 `src/core/auth/*`、`src/app/(dashboard)`）
     - 问题描述：说明问题是什么、为何重要
     - 改进建议：给出现实可行的改法或重构方向

3. 全局改进建议
   - 提出可在多个模块中统一实施的改进（例如统一错误处理模型、统一请求封装、完善 TS 类型策略等）。

4. 优先级行动清单
   - 将上述改进整理为 5–10 条**可执行任务**（task），按优先级排序。
   - 每条包含：简短标题 + 目标模块 + 期望收益。

5. 架构相关的附加观察（可选）
   - 如在代码巡检中发现明显架构性问题，简要列出，并建议使用 `repo-architecture-review` 做专门分析。
