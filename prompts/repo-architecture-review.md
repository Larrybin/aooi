# Repo Architecture Review - 仓库级架构审查

用于对整个仓库进行架构级别的系统性审查，适配当前项目技术栈和目录结构。

## 使用方法

```bash
/zcf:repo-architecture-review <审查关注点或背景说明，可为空>
```

## 项目技术栈与结构假设

- 前端：TypeScript + React + Next.js App Router，路由/页面位于 `src/app`
- 领域层：业务逻辑（auth、数据库、billing、后台任务等）位于 `src/core`
- 共享层：工具方法、基础类型、跨领域能力位于 `src/shared`
- 其他：`src/themes`、`content`、`public` 提供 UI 主题、文档/营销内容和静态资源
- 构建与运行：`pnpm`、Node.js，包含 Cloudflare 部署脚本（`wrangler` 等）

> 如实际仓库结构与上述假设不一致，要求 AI 以实际仓库为准，并在评审中指出差异及潜在影响。

## 输入内容

在调用本 Prompt 时，应向 AI 提供尽可能多的上下文（可由工具自动注入）：

- 仓库目录结构（至少到 `src/app`、`src/core`、`src/shared` 等关键目录）
- 关键配置文件：
  - `package.json`、`tsconfig.json`、`next.config.*`
  - `.eslintrc*`、`.prettierrc*` 等工程规范
  - 数据库/Cloudflare/部署相关配置（如有）
- 代表性的核心模块示例：
  - `src/app` 下 2–3 个典型路由/布局/接口 handler
  - `src/core` 中的典型领域服务/用例/模型
  - `src/shared` 中的通用工具/类型
- 如有：架构设计文档、README 中的架构说明

## 你的角色

你是一名资深全栈架构师，长期在大型 TypeScript/React/Next.js 项目中实践 SOLID、KISS、DRY、YAGNI 原则。

你的目标：

- 评估当前仓库的整体架构质量、分层合理性、边界清晰度和可演进性
- 指出高优先级的架构风险和技术债，并给出**现实可落地**的改进建议
- 确保项目结构与 Next.js + TypeScript 的最佳实践对齐，并尊重现有约束

## 审查时必须覆盖的维度

1. 分层与模块边界
   - `src/app` / `src/core` / `src/shared` 等目录的职责是否清晰？是否存在职责混淆或循环依赖？
   - UI 层（页面/组件）与领域层（业务逻辑）是否良好分离？是否出现页面直接操作基础设施（如数据库、第三方 SDK）的情况？
   - `shared` 中是否只放“跨领域的可复用能力”，而不是“无处安放的杂项代码”？

2. 依赖方向与耦合度
   - 依赖方向是否符合“上层依赖下层抽象而非实现”的原则（例如 `app → core → infra`）？
   - 是否存在明显的双向依赖、跨层调用、对具体实现的强耦合（如页面直接依赖具体 ORM 实体或 HTTP 客户端）？
   - 是否有合适的抽象边界（接口、端口/适配器模式）以便替换实现（如不同数据库、不同消息通道）？

3. Next.js / React 特有架构点
   - `app` 路由结构是否清晰表达业务信息架构？`layout` / `page` / `route handler` 的职责是否分明？
   - Server Components / Client Components 的边界是否合理？是否存在不必要的客户端组件或水合压力？
   - 数据获取（server actions / route handlers / RSC data fetching）是否集中在合适的位置，而非散落在多处？

4. 领域建模与核心业务模块（`src/core`）
   - 领域模型、服务、用例（use case）是否有清晰的边界与命名？是否存在大量“事务脚本”堆积在控制器/路由层？
   - 复杂业务规则是否被显式建模（如值对象、领域服务），还是被硬编码在控制流中？
   - 是否在避免过度设计的前提下，预留了合理扩展点（策略模式、配置驱动等）？

5. 共享模块与横切关注点（`src/shared` 等）
   - 日志、错误处理、配置管理、鉴权、权限控制、缓存等横切关注点是否有统一封装，还是散落在各处？
   - `shared` 中是否存在重复实现的工具方法/类型，可以合并或抽象？
   - 是否有清晰的 error 类型/结果封装，统一错误边界和返回结构？

6. 数据访问与状态管理
   - 数据访问层（数据库、API 客户端、第三方服务）是否通过清晰接口抽象与上层解耦？
   - 前端状态管理方式（React state/hooks、Context、Zustand、Redux 等）是否与页面复杂度匹配？是否存在全局状态滥用？
   - SSR/CSR/Suspense/缓存策略（如 React 服务器组件缓存、`fetch` 缓存）是否与业务需求和性能目标匹配？

7. 性能与可扩展性
   - 是否存在容易放大为性能瓶颈的架构决策（如 N+1 查询、过重的全局中间件、巨型页面组件）？
   - 在功能增长和团队扩张场景下，当前模块划分和边界设计是否易于扩展和分工协作？

8. 安全与合规（架构视角）
   - 鉴权/授权逻辑是否集中在清晰的边界层，而不是零散分布？
   - 配置（密钥、凭证、环境变量）是否通过统一机制管理，而不是硬编码或随处读取 `process.env`？

9. 工程实践与开发体验
   - 工程工具链（lint、format、type-check、build、test、db migrate、云部署脚本等）是否与项目规模匹配且集成良好？
   - 项目结构是否易于新成员理解和上手？命名与目录划分是否存在明显一致性问题？

> 如任务复杂，鼓励在分析前使用 Context7 查询与 Next.js/TypeScript 架构相关的公开最佳实践，并将结论融入审查建议。

## 输出格式要求

AI 的输出应严格按照以下结构组织：

1. 架构总体评估  
   - 用 3–5 句总结整体架构特点、优势与主要风险点。

2. 高优先级问题（必须改进）  
   - 使用列表列出对可维护性/可扩展性/正确性影响较大的架构问题。  
   - 每条包含：严重等级（`[blocker]` / `[major]` / `[minor]`）、涉及模块/目录、问题描述、推荐的改进方向（务必可落地）。

3. 中低优先级改进建议  
   - 针对分层优化、模块重组、抽象收敛、横切能力统一等提出建议。

4. 长期演进建议（可选）  
   - 在不推翻现有系统的前提下，给出 1–3 条中长期重构或演进路径，帮助项目逐步迈向更清晰的架构。
