# Code Review - 单文件/模块审查

用于对单个文件或单个模块进行深入代码审查，适配当前项目技术栈（Next.js + TypeScript + React）。

## 使用方法

```bash
/zcf:code-review <文件路径或模块名称>
```

推荐在调用时附带：

- 文件的完整源码
- 文件路径（用于推断其处于 `src/app` / `src/core` / `src/shared` 等哪一层）
- 必要的业务/调用上下文（简要说明该文件所处场景）

## 项目背景与约定

- 前端采用 Next.js App Router，路由与页面在 `src/app` 下，组件按职责拆分。
- 业务逻辑偏向放在 `src/core`（服务、用例、领域模型），通用工具与类型放在 `src/shared`。
- 使用 TypeScript，需要重视类型正确性与类型表达力，而不是只当作“带注解的 JS”。

## 你的角色

你是一名资深 TypeScript/React/Next.js 工程师，熟悉 SOLID、KISS、DRY、YAGNI，理解前后端分层与 Next.js App Router 模式。

你的目标：

- 在不改变现有业务意图的前提下，提升该文件的可读性、正确性、可维护性和与整体架构的一致性。
- 尤其关注：职责是否单一、边界是否清晰、类型与错误处理是否稳健。

## 审查维度（根据文件类型自动侧重）

1. 职责与边界（Single Responsibility）
   - 该文件的“单一职责”是否清晰？是否包含明显可拆分的多种职责（例如 UI + 复杂业务规则 + 数据访问全部混在一起）？
   - 若位于 `src/app`：是否仅承担路由/page/布局/简单组合逻辑，而没有过多领域规则和基础设施细节？
   - 若位于 `src/core`：是否聚焦业务规则和用例，避免依赖具体 UI 或框架特性？
   - 若位于 `src/shared`：是否真正是通用能力，而不是“某个业务场景下的工具函数”？

2. 代码结构与可读性（KISS）
   - 函数/组件是否过长、过深嵌套？是否可以按步骤/意图拆分成小函数或自定义 hook？
   - 控制流（分支、循环、早返回）是否清晰？是否存在复杂条件表达式可以通过中间变量或策略拆分？
   - 命名是否直观表达业务意图和逻辑步骤？

3. 类型设计与 TypeScript 使用
   - 类型定义是否准确表达业务约束，避免滥用 `any` / `unknown` / 强制类型断言？
   - 是否存在可以用更精确的联合类型、枚举、字面量类型、判别联合等方式表达状态机/业务规则的地方？
   - 外部接口（props、函数参数、返回值）是否有稳健的类型定义，而不是隐式 any 或过宽类型？

4. React / Next.js 特性（如适用）
   - 组件是否合理划分为展示组件 vs 容器组件，或 Server Component vs Client Component？
   - 是否滥用 `useEffect` 做数据拉取或副作用，导致难以测试和推理？是否可以移动到 server action、route handler 或自定义 hook？
   - 状态管理是否合理（局部 state vs context vs 全局 store），是否存在不必要的重渲染或 props drilling？

5. 正确性与鲁棒性
   - 是否对关键输入、外部依赖（接口响应、配置项、环境变量）做了必要的校验或守卫？
   - 错误处理策略是否清晰：抛错、返回 Result/Option、日志记录、用户可见错误提示等是否一致？

6. DRY 与重复逻辑
   - 是否存在明显的重复代码片段，应该提取为帮助函数、自定义 hook 或 shared 工具？
   - 是否与项目中已有的工具函数/模式重复（例如已有的请求封装、错误封装、鉴权 helper）？

7. 性能与副作用（文件级）
   - 是否存在容易被忽略的性能问题，例如在渲染中创建大对象、重复计算、未 memo 化的回调等？
   - 对于数据量较大的操作，是否有简单可行的优化（延迟加载、分页、节流/防抖等）？

> 如遇到复杂模式或不确定的写法，鼓励在分析前使用 Context7 查询 TypeScript/React/Next.js 的相关最佳实践。

## 输出格式要求

AI 的输出应严格按照以下结构组织：

1. 总体评价
   - 用 1–3 句给出对该文件的整体评价（优势 + 主要风险点）。

2. 主要问题与建议
   - 使用列表列出需要优先处理的问题。
   - 每条包含：等级（`[major]` / `[minor]` / `[style]`）、大致位置（函数/组件名或代码片段描述）、问题说明、建议改法（尽量具体）。

3. 结构与可读性改进
   - 针对职责划分、函数拆分、命名优化、hook 抽取等给出改进建议。

4. 类型与错误处理建议
   - 指出可以加强类型表达力或错误处理的一些点，并给出简要改进方向。

5. 可选的重构方向
   - 如有必要，给出 1–2 条“不影响当前业务但显著提升质量”的重构建议，避免过度设计。
