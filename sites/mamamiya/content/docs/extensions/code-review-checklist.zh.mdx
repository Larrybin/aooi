> 目标：把“必须问的问题”和“必须拒绝的情况”前置，让审查更一致、可执行、可回溯。

## 0. 使用方式（推荐流程）

- 审查顺序：**边界与契约 → 安全与数据 → 逻辑正确性 → 性能 → 可维护性**。
- 要求提交者在 PR 描述里补齐“必问问题”的答案（见文末模板），审查者只需要验证证据是否成立。
- 所有结论都应落到可定位的证据：文件路径/函数名/截图/日志/trace，而不是主观判断。

## 1. 架构与边界（强制项）

### 1.1 Server / Client Component 边界正确

- [ ] 能用 Server Component 的地方不写 `use client`。
- [ ] Client Component 仅用于：交互（click / input）、本地状态、浏览器 API（含 `useEffect`）。
- [ ] 避免“页面级 client 化”（`page.tsx` / `layout.tsx` / `template.tsx` 轻薄）。

**Reject 标准（直接拒绝）**

- [ ] 因一个按钮、动画、轻量交互导致整页/整段路由 `use client`。
- [ ] 将鉴权/权限/计费等安全关键判断放在 Client 侧“决定是否能做”。

### 1.2 模块职责清晰（UI 不碰数据源）

- [ ] UI 层（组件/页面）不直接：`fetch` 核心数据、调用 ORM/SDK、拼装权限规则。
- [ ] 数据与业务逻辑集中在服务端边界：`src/core/**`、Route Handlers、Server Actions（按项目约定）。
- [ ] 共享能力下沉到 `src/shared/**`（类型/工具/基础组件），避免跨层引用形成环依赖。

## 2. TypeScript（强制项）

### 2.1 类型真实表达业务

- [ ] 避免 `any`；需要逃逸时用 `unknown` + 缩小（narrowing）并写清楚边界。
- [ ] 避免不必要的 `as`；如果必须断言，说明原因并把断言封装在边界层。
- [ ] 类型表达“意图”（领域约束/单位/枚举语义），而不是为了过编译拼结构。

### 2.2 前后端边界类型明确（稳定契约）

- [ ] Server Action 入参/出参有显式类型（不要让调用方猜）。
- [ ] API / Action 返回是稳定契约：字段含义、可空性、错误结构明确。

**推荐实践**

- [ ] zod schema → `z.infer`：校验与类型同源（Single Source of Truth）。
- [ ] 输入校验只做一次：在“入口边界”完成（Action/Route Handler），内部逻辑不重复校验但也不信任外部。

### 2.3 是否滥用可选属性

- [ ] `foo?: string` 真的可空（业务允许缺失且调用方能处理）。
- [ ] 不把“不确定性”随意推给调用方：优先在服务端收敛为明确状态/枚举。

## 3. 路由与页面组织（App Router）

### 3.1 `page.tsx` 足够薄

- [ ] 只负责：数据聚合、组件组合、路由级配置（metadata 等）。
- [ ] 避免复杂条件分支/业务判断/权限规则堆叠在页面文件里。

**Reject 标准**

- [ ] `page.tsx` / `layout.tsx` 里存在“资源级权限判断/订单状态机/计费规则”等核心业务流程。

### 3.2 `layout` / `template` 使用克制

- [ ] `layout` 只放真正共享 UI（导航、外壳、providers）。
- [ ] 避免把动态数据/state 放进 `layout` 造成全树重渲染或缓存语义混乱。

## 4. 数据获取 & Server Actions（高风险区）

### 4.1 数据获取位置合理

- [ ] 核心数据在 Server Component / Route Handler / Server Action 获取与聚合。
- [ ] Client Component 只消费数据，不“回源”拼装关键数据。

### 4.2 `fetch` 缓存策略明确（必须说清时效性）

- [ ] 明确声明 `cache: 'no-store'` 或 `revalidate`（避免默认缓存导致“看起来实时但其实不实时”）。
- [ ] 有失效策略：需要时结合 `revalidateTag` / `revalidatePath`（按项目约定）。

**必问问题**

- [ ] 这条数据的“时效性”是多少（秒/分钟/小时）？
- [ ] 谁负责触发失效（用户操作/定时/后台事件）？失败时会怎样？

### 4.3 Server Action 安全

- [ ] 入口做 schema 校验（zod）。
- [ ] 入口做身份认证 + 授权校验（资源级别，而不是仅角色）。
- [ ] 不信任客户端传参（金额、userId、tenantId、权限位等必须由服务端决定/校验）。

**Reject 标准**

- [ ] 以客户端提交的关键字段作为最终事实（例如 `amount`、`role`、`isAdmin`、`userId`）。

## 5. 安全（强制项）

### 5.1 Web 基础安全

- [ ] CSRF：对 Cookie 会话 + 有副作用的操作，具备可验证的防护策略（同站策略/CSRF token/双重提交等其一且能解释）。
- [ ] Rate limit / 防刷：登录、验证码、找回密码、下单/回调等关键入口具备限流或滥用防护（按项目约定）。
- [ ] XSS：不渲染未消毒的用户输入；需要渲染 HTML 时有明确的 sanitize 白名单策略。
- [ ] 文件上传：类型/大小/内容（MIME/签名）校验，文件名与路径不信任客户端输入。

### 5.2 多租户/资源边界（如适用）

- [ ] 服务端查询强制带上 `tenantId/orgId` 等隔离条件（不能依赖前端“只展示自己的”）。
- [ ] 资源级授权（谁能读/写哪个资源）有统一入口与可复用策略，避免散落 `if`。

### 5.3 敏感信息与日志

- [ ] 不把 secret 写入 `NEXT_PUBLIC_*`，不在 Client 组件中访问私密环境变量。
- [ ] 日志/错误上报不包含令牌、支付信息、隐私数据；对外错误信息不暴露内部实现细节。

### 5.4 数据库安全（RLS，如适用）

- [ ] 若使用 Supabase/PostgREST 暴露 `public` schema：相关业务表启用 RLS（deny-by-default），并仅对需要暴露的表/操作添加最小 policy。
- [ ] 新增 `public` 表（或将表迁入 `public`）时：同一批迁移包含 `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;`（避免 Supabase linter 0013 回归）。
- [ ] PR 描述说明：是否存在客户端直连/外部暴露；如有，必须附 policy 设计与最小验证口径（SQL/用例）。

**Reject 标准（直接拒绝）**

- [ ] 计划通过 PostgREST/客户端直连访问 `public` 表，但未启用 RLS 或未提供对应 policy 与验证方式。
- [ ] 将具备高权限（如 `service_role`/bypass RLS）的凭据暴露到客户端或写入 `NEXT_PUBLIC_*`。

## 6. Client Component（质量项）

### 6.1 状态最小化

- [ ] state 只承载“必须在客户端变”的信息，派生数据用 `useMemo` 或直接计算。
- [ ] 能 Server 化的状态不留在客户端（例如：权限、计费、数据裁剪规则）。

### 6.2 `useEffect` 克制使用

- [ ] 只做副作用（订阅、埋点、与浏览器 API 交互），不在 effect 里写业务流程。
- [ ] effect 依赖数组正确，不靠禁用 lint 规则“糊过去”。

### 6.3 重渲染风险

- [ ] props 不频繁创建新对象/函数；必要时用 `useMemo`/`useCallback`，或调整组件边界。
- [ ] 不为“理论优化”增加复杂度：先证明热点再优化（YAGNI）。

## 7. Tailwind CSS（定制关注点）

### 7.1 类名可维护

- [ ] 避免超长 class 串；重复组合抽到组件或 `cva/clsx`。
- [ ] 条件样式集中管理，避免在 JSX 中写复杂样式判断污染业务逻辑。

### 7.2 设计系统一致性

- [ ] 遵循既定 spacing / color / font 约定（tokens）。
- [ ] 避免引入“魔法值”（尤其是任意 `px`、`#hex`）破坏一致性。

## 8. 性能（Next + Tailwind + TS 交叉点）

### 8.1 客户端 JS 体积

- [ ] 不引入不必要的大型 UI 库；大型逻辑尽量留在服务端或按需加载。
- [ ] 任何新增依赖都回答：为什么必须？是否可替代？对首屏/交互的影响是什么？

### 8.2 图片与字体

- [ ] 图片用 `next/image`，合理配置 `sizes` / `priority`，避免 CLS。
- [ ] 避免使用运行时 `<img>` 的第三方图片组件；统一封装 `next/image`，且 `fill` / 响应式场景必须提供 `sizes`。
- [ ] 字体加载策略明确（子集/预加载/回退），避免布局抖动。

### 8.3 Streaming / Suspense

- [ ] Suspense 边界清晰，避免无控制嵌套导致瀑布或难以排障。

## 9. 错误处理与用户体验

- [ ] 路由级 `error.tsx` / `loading.tsx`（如适用）齐备且符合体验预期。
- [ ] 系统错误与业务错误区分清楚：对用户可读、对排障可定位。
- [ ] Loading/骨架屏稳定、不抖动；失败态有明确下一步（重试/反馈/引导）。

## 10. 环境变量与配置（强制项）

- [ ] `NEXT_PUBLIC_*` 仅用于可公开信息；私密 key 只存在于服务端。
- [ ] 配置集中在统一模块（例如 `src/config/**`），不散落在组件中。
- [ ] 默认值与校验策略明确（缺失时 fail-fast 还是降级）。

## 11. 可测试性与长期维护

- [ ] 业务逻辑从组件中抽离到可复用函数/服务（便于单测与复用）。
- [ ] Server Action 具备可单测的核心逻辑（至少能在不启动 UI 的情况下跑）。
- [ ] 新需求不需要“大改结构”才能接入；若需要，说明当前抽象不足并给出最小重构方案。

## 附：PR 描述模板（可复制）

```md
## 变更概述

- 这次改了什么？为什么必须现在改？

## 影响范围

- 影响的路由/模块：
- 是否涉及：认证/权限/计费/订阅/数据迁移（是/否）

## 缓存与时效性（如有数据变更）

- 这条数据的时效性：\_\_（例如 0s/10s/5m/1h）
- 缓存策略：`cache: ...` / `revalidate: ...` / `revalidateTag|Path: ...`
- 失效触发：\_\_（用户操作/后台事件/定时）

## 安全说明（如有副作用操作）

- 输入校验：\_\_（schema 链接/文件）
- 鉴权与授权：\_\_（在哪里做的？资源级规则是什么？）
- 幂等与并发：\_\_（重复提交/重放会怎样？）

## 回滚方案

- 如何快速回滚/降级：\_\_

## 证明与验证

- 关键截图/录屏：
- 本地验证：`pnpm lint` / `pnpm format:check` / `pnpm build`
```
